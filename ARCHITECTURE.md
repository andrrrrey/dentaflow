# DentaFlow — описание проекта и архитектуры

> Документ написан простым языком для тех, кто хочет понять, **что это за система, из чего она состоит и почему она устроена именно так**. Подходит и разработчику, который только пришёл в проект, и руководителю, который хочет понять «как это работает под капотом».

---

## 1. Что это за продукт и зачем он нужен

**DentaFlow** — это интеллектуальный веб-дашборд для управления стоматологической клиникой.

Ключевая идея: клиника уже работает в медицинской информационной системе (МИС) **1Denta**. DentaFlow **не заменяет** её, а надстраивается сверху и добавляет то, чего в МИС нет:

- единый коммерческий и управленческий слой (CRM, аналитика, маркетинг);
- ИИ-подсказки администраторам и руководителю;
- единый центр всех коммуникаций с пациентами;
- автоматические задачи и напоминания.

Проще говоря: **1Denta — это «бухгалтерия» приёмов и медкарт, а DentaFlow — это «мозг продаж и управления» поверх неё.**

### Масштаб и контекст
- 1 клиника, до ~10 сотрудников.
- База ~17 000 пациентов (важная цифра — она объясняет многие технические решения, см. ниже).
- Клиника физически находится в Улан-Удэ (часовой пояс Москва +5) — это влияет на расписание фоновых задач.
- Деплой на VPS (Ubuntu) в облаке Timeweb.

### Из чего состоит продукт (модули)
1. **Дашборд руководителя** — ИИ-аналитика, воронка пациентов, загрузка врачей, рейтинг администраторов, ежедневный отчёт в Telegram.
2. **Центр коммуникаций + CRM-воронка** — единая лента входящих (Telegram, Max/VK, сайт, телефония Novofon), канбан-воронка сделок, карточка пациента «360°», ИИ-подсказки.
3. **Дополнительные разделы**, выросшие со временем: маркетинг, сегменты базы пациентов, контроль качества звонков и скриптов, задачи, справочники, мотивация персонала (баллы/награды).

## 2. Стек технологий (и почему именно он)

| Слой | Технология | Почему выбрано |
|------|-----------|----------------|
| Фронтенд | React 18 + TypeScript + Vite | Стандарт для интерактивных SPA; TypeScript ловит ошибки на этапе написания кода; Vite даёт быструю сборку и горячую перезагрузку. |
| Стили | TailwindCSS | Быстрая вёрстка прямо в разметке, единый дизайн без «зоопарка» CSS-файлов. |
| Серверные данные | TanStack React Query | Кэширование запросов, автообновление, меньше ручного кода для загрузки данных. |
| Состояние клиента | Zustand | Лёгкий стор для авторизации, UI и уведомлений — проще, чем Redux. |
| Бэкенд | FastAPI (Python 3.11) | Асинхронный, быстрый, автоматическая документация API, удобная валидация через Pydantic. |
| ORM / БД | SQLAlchemy 2.0 (async) + PostgreSQL 15 | Зрелая связка; PostgreSQL умеет JSONB и массивы, что важно для хранения «сырых» данных из 1Denta и ИИ-результатов. |
| Миграции | Alembic | Версионирование схемы БД, безопасное обновление продакшена. |
| Кэш/очереди/realtime | Redis 7 | Одновременно брокер для Celery, кэш и шина Pub/Sub для WebSocket. |
| Фоновые задачи | Celery (worker + beat) | Тяжёлые и периодические операции вынесены из веб-процесса. |
| Telegram-бот | aiogram 3 | Современный асинхронный фреймворк для ботов. |
| ИИ | OpenAI (GPT-4o / gpt-4o-mini) | Аналитика, подсказки, классификация пациентов и обращений. |
| Прокси / SSL | Nginx | Раздача статики фронтенда, SSL, проксирование на API. |
| Упаковка | Docker + docker-compose | Одинаковое окружение локально и на проде, простой деплой. |

**Общий принцип выбора:** проверенные, асинхронные инструменты, которые хорошо работают вместе и не требуют большой команды для поддержки.

## 3. Архитектура в целом

```
┌─────────────────────────────────────────────┐
│            Браузер (React + TS)              │
│         SPA, общается по HTTPS + WebSocket    │
└───────────────────────┬───────────────────────┘
                        │
┌───────────────────────▼───────────────────────┐
│                   NGINX                        │
│   SSL, отдаёт статику фронта, проксирует /api  │
└───────────────────────┬───────────────────────┘
                        │
┌───────────────────────▼───────────────────────┐
│              FastAPI (бэкенд)                  │
│   REST API + WebSocket, JWT-авторизация        │
└───┬───────────┬──────────────────┬────────────┘
    │           │                  │
┌───▼────┐  ┌───▼────┐   ┌─────────▼──────────────┐
│Postgres│  │ Redis  │   │   Celery workers       │
│(данные)│  │(кэш/   │   │ - синк с 1Denta        │
│        │  │ очередь│   │ - сегменты базы (ИИ)   │
│        │  │ /pubsub│   │ - напоминания/задачи   │
└────────┘  └────────┘   │ - ежедневный отчёт     │
                         └────────────────────────┘
                                  │
┌──────────────────────────────────▼─────────────┐
│   Внешние сервисы: 1Denta, Novofon (телефония), │
│   OpenAI, Telegram, Max/VK                       │
└─────────────────────────────────────────────────┘
```

### Почему именно такое разделение
- **Фронт и бэк разделены** (SPA + REST). Это позволяет независимо развивать интерфейс и логику, а также легко открыть API для будущих интеграций.
- **Тяжёлая работа — в Celery, а не в вебе.** Веб-процесс обязан отвечать пользователю быстро. Синхронизация с 1Denta (≈150 секунд на полный прогон) или ИИ-анализ 17 000 пациентов не должны «вешать» интерфейс — поэтому они вынесены в фоновые задачи.
- **Redis — универсальный «клей».** Одновременно служит брокером задач Celery, кэшем и шиной событий для real-time обновлений в браузере (через WebSocket).

## 4. Структура кода

### Бэкенд (`backend/app/`)

```
main.py            — точка входа FastAPI, подключение всех роутеров
config.py          — все настройки через переменные окружения (pydantic-settings)
database.py        — асинхронный движок SQLAlchemy
dependencies.py    — DI: получить БД, текущего пользователя, проверка роли

models/      — таблицы БД (ORM): patient, deal, communication, user, task,
               appointment, patient_segment, knowledge_base, и т.д.
schemas/     — Pydantic-схемы (валидация входящих/исходящих данных API)
routers/     — HTTP-эндпоинты, сгруппированные по темам (auth, patients,
               deals, dashboard, communications, marketing, ai, ws, ...)
services/    — бизнес-логика и интеграции (one_denta, novofon, ai_service,
               telegram_bot, segments_service, dashboard_service, ...)
tasks/       — фоновые Celery-задачи (sync_1denta, segments, auto_tasks,
               daily_report, bot_reminders, alerts, ai_insights)
utils/       — вспомогательное: security (JWT, хэш паролей), seed
alembic/     — миграции БД (20+ версий — история развития схемы)
```

**Принцип слоистости** (важное архитектурное решение):
- **routers** — только приём запроса, проверка прав, вызов сервиса, возврат ответа. Тонкий слой.
- **services** — вся «умная» логика и общение с внешним миром.
- **models/schemas** — данные и их валидация.

Зачем так: логику можно переиспользовать и из роутера, и из Celery-задачи, и тестировать отдельно. Роутеры остаются простыми и читаемыми.

### Фронтенд (`frontend/src/`)

```
api/         — клиенты к бэкенду (по одному файлу на раздел: patients.ts,
               deals.ts, dashboard.ts, ...), единый axios-client.ts
pages/       — страницы под маршруты (Dashboard, Pipeline, Communications,
               PatientCard, Marketing, ...)
components/  — переиспользуемые блоки (layout, ui, pipeline, schedule, ...)
store/       — Zustand-сторы (authStore, uiStore, notificationStore, ...)
hooks/       — useAuth, useWebSocket
types/       — общие TypeScript-типы
```

Маршрутизация — `react-router-dom`. Каждая страница обёрнута в `ProtectedRoute` (проверка авторизации) и `AppLayout` (общий каркас с навигацией). У части страниц есть ограничение по ролям через `denyRoles` (например, администратору закрыты «Финансы», «Отчёты», «Сотрудники», «Настройки»).

## 5. Ключевые механизмы и почему они так сделаны

### 5.1. Интеграция с 1Denta — «источник правды» о приёмах

1Denta — внешняя МИС с REST API (SQNS CRM Exchange). DentaFlow забирает оттуда пациентов, расписание, приёмы, справочники врачей и услуг.

Технические решения и их причины:
- **JWT-токен кэшируется в памяти процесса и автоматически обновляется при 401.** Не дёргаем авторизацию на каждый запрос — экономим время и нагрузку.
- **«Сырые» данные из 1Denta хранятся в поле `raw_1denta_data` (JSONB).** Если завтра понадобится поле, которое сегодня не разобрали, его не нужно тянуть из 1Denta заново — оно уже лежит в базе.
- **Несколько режимов синхронизации с разной частотой** (см. расписание ниже): часто — только ближайшие приёмы (быстро и дёшево), редко (ночью) — полная синхронизация всего.

### 5.2. Фоновые задачи Celery и расписание (beat)

Все периодические операции вынесены в Celery. Расписание (часовой пояс — Москва):

| Задача | Частота | Зачем |
|--------|---------|-------|
| Синк ближайших приёмов | каждые 5 мин | Расписание всегда актуально, но нагрузка минимальна. |
| Синк справочников (врачи/услуги) | раз в час | Меняются редко. |
| Полная синхронизация | ночью 02:30 | Тяжёлая (~150с) — делается, когда никто не работает. |
| Проверка «зависших» лидов | каждые 5 мин | Не упустить пациента, с которым давно не связывались. |
| Обновление ИИ-инсайтов | раз в час | Аналитика на дашборде свежая. |
| Ежедневный отчёт в Telegram | 20:00 | Итоги дня руководителю. |
| Напоминания о приёме | каждые 15 мин | Своевременные сообщения пациентам. |
| Создание задач на обзвон (подтверждение завтрашних приёмов) | 03:00 МСК = 08:00 в Улан-Удэ | Задачи готовы ровно к открытию клиники — у админа весь день на обзвон. |
| Создание задач-follow-up по вчерашним | 03:05 МСК | Догнать тех, кто был вчера. |
| Деактивация просроченных задач | 00:05 | Чистка перед генерацией новых. |

**Очень важное решение — разделение очередей Celery.** Воркеры слушают очереди `default, ai, sync`, а тяжёлый пересчёт сегментов базы вынесен на **отдельную очередь `segments` со своим воркером** (`concurrency=1`). Причина прямо описана в коде: иначе ночная полная синхронизация и синк приёмов каждые 5 минут «съедали» единственный воркер, и пересчёт сегментов навсегда зависал в статусе «в очереди». Изоляция гарантирует, что эти процессы не мешают друг другу.

### 5.3. Сегменты базы пациентов и ИИ — самый «хитрый» модуль

Это сохранённые умные списки пациентов, например: «незавершённое лечение», «несостоявшиеся консультации», «пора на гигиену». Часть из них формируется не простым SQL-фильтром, а **ИИ-вердиктом по истории визитов каждого пациента**.

Проблема: пациентов ~17 000. Прогонять всех через GPT каждый раз — дорого и медленно. Решения:
- **Кэш результата по «отпечатку» (fingerprint).** Для каждого пациента считается хэш его истории. Если история не изменилась — повторного обращения к OpenAI не происходит, берётся сохранённый результат. (Поля `ai_analysis_fingerprint`, `treatment_ai_fingerprint` в модели пациента.)
- **Более дешёвая модель и параллелизм для массового анализа** (`gpt-4o-mini`, до 15 параллельных запросов) — отдельно от точечного анализа, где используется `gpt-4o`.
- **Результаты участников сегмента сохраняются в БД**, чтобы при открытии списка не пересчитывать заново.
- Логика списков «привязана к реальным фактам визитов», а не к «рыхлым» ИИ-флагам — чтобы не попадать в список ошибочно (это видно по истории коммитов: специально чинили логику сегментации).

### 5.4. ИИ-сервис с режимом «моков»

`ai_service.py` умеет работать **без ключа OpenAI**: в режиме разработки методы возвращают реалистичные данные на русском. Это позволяет показывать и разрабатывать дашборд, не тратя деньги на API и не требуя ключа. Также есть аккуратная обработка ошибки «закончилась квота/баланс» (`OpenAIQuotaError`) — чтобы система не падала, а корректно сообщала о проблеме.

### 5.5. Реал-тайм обновления (WebSocket + Redis Pub/Sub)

Когда приходит новое сообщение от пациента или меняется сделка, интерфейс обновляется без перезагрузки. Механизм:
1. Сервис или Celery-задача публикует событие в канал Redis (`dentaflow:events`).
2. Бэкенд через WebSocket рассылает событие подключённым браузерам.

Важная деталь: **если Redis недоступен (например, локальная разработка без Docker), сервис «мягко деградирует»** — просто работает без realtime, не падая. Это снижает порог входа для разработчика.

### 5.6. Аутентификация и роли

- **JWT с двумя токенами:** короткоживущий access (15 мин) и долгоживущий refresh (30 дней). Короткий access снижает риск при утечке токена, refresh избавляет пользователя от частых перелогинов.
- Пароли хэшируются (`bcrypt`).
- **Роли:** `owner | manager | admin | marketer`. Проверка прав есть и на бэкенде (`role_required`), и на фронтенде (`denyRoles` в маршрутах). Дублирование намеренное: фронт прячет лишнее из UI для удобства, но реальную защиту обеспечивает бэкенд.

### 5.7. Каналы коммуникаций

Единая модель `Communication` хранит обращения из всех каналов (Telegram, Max/VK, сайт, звонки Novofon) в одном виде: канал, направление (входящее/исходящее), тип, содержимое, статус, приоритет. Поверх — ИИ-поля: теги, краткое содержание, рекомендованное следующее действие. Это и есть «единая лента входящих»: администратор видит всё в одном месте, а не в пяти разных приложениях.

## 6. Как это разворачивается (деплой)

Два docker-compose файла:

**Локально (`docker-compose.yml`)** — Postgres, Redis, backend (с автоперезагрузкой), один celery-worker, celery-beat, фронт через `vite dev`. Всё с монтированием кода для удобной разработки.

**Прод (`docker-compose.prod.yml`)** — отличия отражают зрелость системы:
- Фронт собирается в статику и отдаётся через **Nginx** (SSL, сертификаты Let's Encrypt).
- Backend стартует с `alembic upgrade head` — миграции применяются автоматически при деплое.
- **Несколько Celery-воркеров:** общий (`default,ai,sync`, concurrency=4) + **отдельный для сегментов** (`segments`, concurrency=1) — то самое разделение очередей.
- Отдельный сервис `telegram_bot`.
- `celery_beat` как планировщик.
- Кэш `node_modules` фронта в volume — для быстрых пересборок.

Скрипты `deploy.sh` / `update.sh` автоматизируют выкладку. Конфигурация полностью через `.env` (`.env.example` — шаблон со всеми ключами: 1Denta, Novofon, Telegram, OpenAI, Max/VK, почта).

## 7. Сводка по принятым решениям (кратко «почему так»)

| Решение | Причина |
|---------|---------|
| Надстройка над 1Denta, а не замена | Клиника уже работает в МИС; дешевле и безопаснее добавить слой, чем мигрировать. |
| Async везде (FastAPI, SQLAlchemy, httpx) | Много обращений к внешним API — асинхронность не блокирует процесс на ожидании сети. |
| Celery + отдельные очереди | Тяжёлые задачи не должны мешать ни вебу, ни друг другу (особенно синк vs пересчёт сегментов). |
| JSONB для сырых данных и ИИ-результатов | Гибкость: новые поля без миграций, кэш ИИ прямо в записи пациента. |
| Fingerprint-кэш для ИИ | Экономия денег и времени на 17k пациентов — не анализировать неизменившихся. |
| Дешёвая модель для массового анализа | Баланс цена/качество: gpt-4o-mini для объёма, gpt-4o для точечных задач. |
| Моки в ИИ-сервисе | Разработка и демо без ключа и расходов. |
| Мягкая деградация Redis | Низкий порог входа для локальной разработки. |
| Двойная проверка ролей (фронт+бэк) | Удобство UI + реальная безопасность на сервере. |
| Расписание задач под часовой пояс клиники | Задачи на обзвон готовы ровно к открытию клиники в Улан-Удэ. |
| Слоистость (router → service → model) | Переиспользование логики, тестируемость, простые роутеры. |

## 8. С чего начать изучение кода

1. `backend/app/main.py` — какие разделы API вообще есть.
2. `backend/app/config.py` и `.env.example` — какие внешние сервисы подключены.
3. `backend/app/tasks/celery_app.py` — что и когда происходит в фоне (отличная карта «жизни» системы).
4. `backend/app/services/one_denta.py` — как устроен главный источник данных.
5. `backend/app/services/segments_service.py` + `tasks/segments.py` — самый сложный и интересный модуль (ИИ + кэш + очереди).
6. `frontend/src/App.tsx` — карта всех страниц и прав доступа.
