# AI-Робот для исходящих звонков — описание проекта

Этот документ объясняет простым языком, как устроен проект, из каких частей он
состоит, как они взаимодействуют и — главное — **почему** в разработке приняты
именно такие решения. Его можно читать без глубокого знания кода.

---

## 1. Что это вообще такое

Это облачный **голосовой робот**, который сам ведёт исходящие телефонные
разговоры на русском языке. Конкретный сценарий, зашитый в проект, — холодный
звонок от компании «РусЭнергоСтрой» по поводу обязательных проверок электросетей:
робот пробивается через секретаря, выходит на ответственного (ЛПР — «лицо,
принимающее решения»), квалифицирует клиента (узнаёт сроки работ, как заключают
договор, берёт контакт) и завершает разговор.

Технически робот делает три вещи по кругу:

1. **Слушает** человека и переводит речь в текст (ASR — распознавание речи).
2. **Думает**, что ответить (логика диалога + ИИ).
3. **Говорит** — превращает текст обратно в голос (TTS — синтез речи).

Всё это работает в реальном времени через браузер: есть демо-страница, где можно
поговорить с роботом голосом через микрофон, и админ-панель для настройки.

## 2. Технологический стек и почему он такой

| Технология | Где используется | Почему выбрана |
|---|---|---|
| **Python 3.11 + FastAPI** | Весь бэкенд | FastAPI — асинхронный, быстрый, со встроенной валидацией (Pydantic) и автодокументацией (Swagger). Асинхронность критична: робот одновременно ждёт ответы от нескольких внешних API. |
| **Uvicorn** | Запуск приложения | Лёгкий ASGI-сервер, штатная пара для FastAPI. |
| **WebSocket** | Поток аудио в реальном времени | Обычный HTTP-запрос не годится для непрерывного двустороннего потока звука. WebSocket держит постоянное соединение, по которому летят аудио-чанки туда и обратно. |
| **Yandex SpeechKit** | TTS (синтез) и ASR (распознавание) | Российский облачный сервис, хорошо работает с русской речью, не требует своей инфраструктуры под ML-модели. |
| **Yandex GPT (Foundation Models)** | Классификация реплик и генерация ответов | Тоже российское облако, недорогая модель `yandexgpt-lite` — быстрая (в 3–5 раз быстрее «pro»), чего достаточно для классификации коротких фраз. |
| **SaluteSpeech (Sber)** | Альтернативный синтез голоса | Второй провайдер TTS — даёт выбор голосов и страховку, если голоса Яндекса не подходят. |
| **ChromaDB** | База знаний и слой правок | Встраиваемая (embedded) векторная БД. Не нужен отдельный сервер — работает прямо в процессе, хранит эмбеддинги на диске. Позволяет «семантический» поиск по смыслу, а не по точным словам. |
| **Redis** | Заложен в инфраструктуру | Подготовлен в docker-compose для будущего хранения состояния/очередей (в текущем коде состояние звонков пока в памяти процесса). |
| **Docker + Docker Compose** | Развёртывание | Один раз описали окружение — запускается одинаково на любом сервере. Compose поднимает сразу робота, Caddy и Redis. |
| **Caddy** | Обратный прокси + HTTPS | Браузер не даёт доступ к микрофону без HTTPS. Caddy автоматически получает SSL-сертификат от Let's Encrypt — не надо вручную возиться с сертификатами. |
| **Loguru** | Логирование | Проще стандартного `logging`, удобная ротация файлов и цветной вывод. |

**Главная идея стека:** не держать у себя тяжёлые ML-модели, а собрать систему из
готовых облачных «кубиков» (распознавание, синтез, GPT). Своя задача проекта —
**оркестрация**: вовремя вызвать нужный сервис и провести разговор по логике скрипта.

## 3. Общая архитектура

```
                            БРАУЗЕР (demo.html / admin.html)
                       микрофон → аудио-чанки      ← голос робота
                                    │   ▲
                                    ▼   │  WebSocket /ws/audio/{call_id}
        ┌───────────────────────────────────────────────────────────┐
        │                     FastAPI (app/main.py)                   │
        │                                                             │
        │   api/routes.py  ── REST + WebSocket эндпоинты              │
        │        │                                                    │
        │        ▼                                                    │
        │   AudioPipeline ── буфер + VAD (определение пауз)           │
        │        │                                                    │
        │   ┌────┴───────────────┬──────────────┬─────────────┐      │
        │   ▼                    ▼              ▼             ▼      │
        │  ASR              DialogueEngine   ScriptV2      TTS /      │
        │ (Yandex)          (v1, гибкий)    (v2, скрипт)  SaluteSpeech│
        │                        │              │                    │
        │                        ▼              ▼                    │
        │                  YandexGPT      KnowledgeBase / Corrections │
        └───────────────────────────────────────────────────────────┘
                                    │
                          CallManager (состояние звонков, история на диск)
```

Код разбит на слои:

- **`app/core/`** — конфигурация (`config.py`) и логирование (`logging.py`).
- **`app/api/routes.py`** — все HTTP- и WebSocket-эндпоинты (входная точка).
- **`app/services/`** — вся бизнес-логика, каждый сервис отвечает за одну вещь.
- **`app/static/`** — фронтенд (две HTML-страницы без фреймворков).
- **`scenarios/`** — YAML-сценарии для гибкого алгоритма v1.

**Почему так:** разделение на слои и отдельные сервисы означает, что каждый
кусок можно менять и тестировать независимо. Например, заменить провайдера
синтеза речи — это правка одного файла `tts.py`, а не всего проекта.

## 4. Как устроен один разговор (поток данных)

Самое важное происходит в WebSocket-обработчике `audio_websocket` в `routes.py`.
Пошагово:

1. **Клиент звонит** — фронтенд вызывает `POST /api/v1/calls/start`, создаётся
   сессия звонка (`CallManager`), робот возвращает приветствие.
2. **Открывается WebSocket** `/ws/audio/{call_id}` — постоянный канал для звука.
3. **Браузер шлёт аудио-чанки** (PCM 8kHz, 16 бит, моно) маленькими кусочками.
4. **`AudioPipeline`** накапливает их в буфер и следит за громкостью (VAD):
   - пока человек говорит — копит;
   - как только наступила **пауза ~1.5 сек** — считает, что реплика закончена;
   - если человек заговорил, **пока робот говорит** — это **перебивание**
     (interrupt), робота прерывают.
5. **ASR** превращает накопленное аудио в текст.
6. Текст уходит в **движок диалога** (v1 или v2 — см. ниже), который решает,
   что ответить и в какую «фазу» перейти.
7. **TTS** синтезирует ответ и **стримит** его обратно в браузер по мере
   генерации — первый кусочек звука уходит, не дожидаясь конца фразы.
8. Цикл повторяется. Когда разговор завершается, **`CallAnalyzer`** через GPT
   составляет саммари и квалифицирует клиента (заинтересован / перезвонить /
   отказ), а звонок сохраняется в JSON-файл.

**Почему стриминг везде, где можно:** в живом разговоре задержка убивает
ощущение естественности. Поэтому:
- ASR-поиск по базе знаний запускается **параллельно** с подготовкой промпта
  (`asyncio.create_task`);
- классификация намерения и генерация ответа в v1 идут **одновременно**
  (`asyncio.gather`) — общее время равно времени самого долгого вызова, а не
  их сумме;
- TTS отдаёт звук **по кускам**, а не целиком.

## 5. Два движка диалога: v1 и v2

Это ключевое архитектурное решение. В системе **два разных алгоритма** ведения
диалога, и пользователь выбирает версию при старте звонка (`algo_version`).

### v1 — гибкий, на основе ИИ (`dialogue_engine.py` + `scenario_engine.py`)

- Сценарий описан в **YAML-файле** (`scenarios/default.yaml`): шаги, и куда
  переходить при `on_positive` / `on_negative` / `on_objection` / `on_unknown`.
- На каждую реплику GPT делает две вещи: **классифицирует намерение** (позитив/
  негатив/возражение/непонятно) и **генерирует ответ** свободным текстом, с
  учётом базы знаний.
- Подходит, когда нужна гибкость и естественность.
- **Минус:** ИИ может «придумать» что-то не то, повториться, поздороваться
  второй раз. Поэтому в промпты добавлены жёсткие запреты («СТОП. Разговор уже
  идёт — приветствие произнесено. ЗАПРЕЩЕНО…»).

### v2 — жёсткий скрипт, ИИ только классифицирует (`script_dialogue_v2.py`)

Это «рабочая лошадка» проекта — самый большой файл (~2100 строк). Идея:

> **ИИ не сочиняет ответы. Все фразы робота заранее написаны человеком**
> (в `script_v2_data.py`). ИИ нужен только чтобы **понять смысл** реплики
> собеседника и выбрать один из заранее заданных «кодов намерения».

Это **конечный автомат** (state machine) с фазами:
`secretary` → `lpr_greeting` → `lpr_main` → `qualification` → `closed`.

Каждая фаза — отдельный обработчик (`_handle_secretary`, `_handle_lpr_main` и
т.д.) с десятками флагов состояния в `V2SessionState` (например,
`secretary_boss_report_asked`, `lpr_far_date_pending`), которые помнят, какой
уточняющий вопрос уже задавали.

**Почему такой подход, а не «просто ИИ»:**

- **Предсказуемость и контроль.** В продажах важно, чтобы робот говорил ровно
  утверждённые юристом/маркетингом фразы. Скрипт = гарантия, что робот не
  ляпнет лишнего и не даст ложных обещаний.
- **Дёшево и быстро.** Классификация короткой фразы в один код — это крошечный
  запрос к лёгкой модели, а не генерация длинного текста.
- **Гибридная надёжность.** Перед обращением к ИИ работает несколько защитных
  слоёв детерминированной логики (без ИИ вообще):
  - **`_KEYWORD_INTENTS`** — «высокосигнальные» фразы (например, «откуда у вас
    наш номер») сразу дают код без вызова GPT. Быстро и не ошибётся.
  - **`TRANSFER_SIGNALS` / `LPR_PICKUP_SIGNALS`** — слова «соединяю»,
    «переведу», «алло», «слушаю» детектируются напрямую, с аккуратным
    исключением отрицаний («не соединяю»).
  - Разбор сроков, месяцев, номеров телефона — регулярками и словарями.
- ИИ-классификатор вызывается **только если детерминированные правила не
  сработали**, с очень низкой температурой (0.05) и ответом в один код.

**Защита от типичных проблем диалога** (тоже важные решения):

- **Детекция зацикливания** (`loop_streak`): если робот дважды подряд выдал один
  и тот же ответ, запускается «восстановление из петли» — отдельный промпт
  просит ИИ найти альтернативную трактовку реплики.
- **Debug неизвестных** (`v2_debug_unknowns/unknowns.jsonl`): если ИИ не смог
  подобрать код (`unknown`), эпизод записывается в файл со всем контекстом —
  чтобы потом дописать скрипт под реальные ответы людей. Это встроенный цикл
  улучшения.
- **Лимит сессий** (`MAX_SESSIONS = 500`): при переполнении удаляется самая
  старая — защита от утечки памяти.

**Вывод по двум движкам:** v1 — это «умно, но рискованно», v2 — «жёстко, но
надёжно». Наличие обоих — осознанный компромисс: можно сравнивать качество и
выбирать под задачу.

## 6. Слой правок ответов (`script_corrections.py`)

Очень практичное решение для **не-программистов**. Заказчик/оператор может
загрузить таблицу (`.xlsx`/`.csv`) из трёх колонок:

1. пример фразы собеседника (триггер),
2. что робот отвечает сейчас (для справки),
3. **как правильно** должен ответить.

Если реальная реплика человека **семантически близка** к триггеру (поиск по
смыслу через ChromaDB, порог cosine-дистанции 0.25) — робот произносит
«правильный» ответ из таблицы **вместо** скриптового. При этом логика
переходов между фазами не ломается — подменяется только текст реплики.

**Почему это ценно:** скрипт можно дорабатывать «на лету», без релиза кода —
просто отредактировав Excel. Это закрывает разрыв между разработчиком и
бизнесом.

## 7. База знаний (`knowledge_base.py`)

- Загружаются документы (`.txt`/`.pdf`/`.docx`), текст режется на
  перекрывающиеся куски (`chunk_size=500, overlap=50`) и кладётся в ChromaDB.
- При диалоге по реплике клиента ищутся **3 самых релевантных по смыслу**
  фрагмента и подмешиваются в промпт GPT как «релевантная информация».
- **Перекрытие кусков** нужно, чтобы важная мысль не разорвалась ровно на
  границе фрагмента.
- Есть **прогрев** (`warmup`) модели эмбеддингов при старте, чтобы первый
  реальный запрос не тормозил из-за «холодного старта» ONNX-модели.

## 8. Конфигурация и устойчивость

- **Все настройки — через `.env`** и Pydantic `Settings` (`config.py`):
  ключи API, голоса, лимиты, пути. Секреты не лежат в коде/Git.
- **`@lru_cache` на `get_settings()`** — настройки читаются один раз и
  переиспользуются (singleton).
- **Персистентные HTTP-клиенты** (`httpx.AsyncClient` в TTS/ASR/GPT) —
  переиспользуют TCP/TLS-соединения, что заметно экономит время на каждом
  вызове облачных API.
- **Таймауты по 15 секунд** на вызовы GPT — чтобы разговор не «завис».
- **Обработка отказов фильтра безопасности GPT** (`SafetyRefusalError`): если
  модель отказалась отвечать на сложный промпт, делается повтор с
  минимальным промптом, а в крайнем случае — фоллбэк на фразу из сценария.
- **Лимит длины текста для TTS** (~230 символов на запрос Яндекса): длинные
  ответы автоматически режутся по границам предложений.
- **Логи разделены** на общий, по звонкам (фильтр по `call`) с ротацией и
  хранением (30/90 дней).

**Общий принцип:** ни один сбой внешнего сервиса не должен ронять разговор —
везде есть try/except и разумный запасной вариант.

## 9. Хранение состояния звонков (`call_manager.py`)

- Активные звонки живут **в памяти процесса** (`dict` + `asyncio.Lock` для
  безопасности при конкуренции).
- Завершённые сохраняются в **JSON-файлы** на диск (транскрипт, саммари,
  статус, длительность).
- Есть лимит **параллельных звонков** (`max_concurrent_calls`).

**Честное замечание об ограничениях:** состояние в памяти означает, что при
перезапуске контейнера активные звонки теряются, и масштабировать на несколько
процессов/воркеров в таком виде нельзя (поэтому в Dockerfile `--workers 1`).
Redis в инфраструктуре уже заложен — это очевидный следующий шаг для выноса
состояния наружу.

## 10. Фронтенд (`app/static/`)

Две самодостаточные HTML-страницы **без фреймворков** (чистый HTML/CSS/JS,
тёмная тема):

- **`demo.html`** — демонстрация голосового контура: захват микрофона
  (`getUserMedia`), конвертация в PCM 8kHz через `AudioContext`, отправка по
  WebSocket, проигрывание ответа.
- **`admin.html`** — панель управления: настройка промптов ИИ, база знаний,
  таблица правок, тест диалога текстом, история звонков. Путь `/test`
  открывает ту же страницу в урезанном режиме «только тест диалога».

**Почему без React/Vue:** для двух страниц фреймворк — лишняя зависимость и
сборка. Один HTML-файл проще развернуть и поддерживать.

## 11. Развёртывание (Docker)

`docker-compose.yml` поднимает три контейнера:

1. **`ai-robot`** — само приложение (FastAPI/Uvicorn), порт 8000, с
   health-check.
2. **`caddy`** — обратный прокси и автоматический HTTPS (нужен для микрофона).
3. **`redis`** — хранилище (задел на будущее).

Данные (логи, записи, сценарии, база знаний, история) вынесены в **volumes**,
чтобы переживать пересборку контейнера. Подробная пошаговая инструкция — в
`DEPLOY.md`.

## 12. Тестирование (`tests/`)

## 13. Итог: какие идеи лежат в основе

1. **Оркестрация, а не своя ML-инфраструктура** — собираем систему из готовых
   облачных сервисов (SpeechKit, YandexGPT, SaluteSpeech).
2. **Скорость через параллелизм и стриминг** — всё, что можно делать
   одновременно, делается одновременно; звук течёт кусками.
3. **Контроль через скрипт (v2)** — для продаж предсказуемость важнее
   «креатива» ИИ; ИИ работает классификатором, а не сочинителем.
4. **Многослойная надёжность** — детерминированные правила перед ИИ, фоллбэки
   на каждом шаге, детекция петель и неизвестных реплик.
5. **Доработка без кода** — таблица правок и YAML-сценарии позволяют менять
   поведение робота, не трогая Python.
6. **Простота развёртывания** — Docker Compose + Caddy: одна команда, и
   автоматический HTTPS.

Главный компромисс, о котором стоит помнить: **состояние пока в памяти**, что
ограничивает масштабирование, — но архитектура уже подготовлена к выносу его в
Redis.
